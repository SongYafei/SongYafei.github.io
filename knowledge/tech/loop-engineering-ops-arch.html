1 范式转换

从 Prompt Engineering 到 Loop Engineering 的演进

2023–2024 · Prompt Engineering
你写好 prompt，LLM 生成输出。交互模式：写 prompt → 看结果 → 改 prompt → 再看。效率完全取决于你 prompt 写得多好。
2024–2025 · Context Engineering
你开始注重视上下文：RAG、记忆系统、项目规则注入。模型输出质量提升，但你仍然是每次对话的"发起者"。
2025–2026 · Harness Engineering
你搭建单个 Agent 的运行环境：工具链、MCP 连接器、安全边界。Agent 能做更多事，但仍然是"你让它做什么它就做什么"。
2026 · Loop Engineering ← 你现在在这里
你不再"催"Agent。你设计一个系统，让 Agent 自己决定做什么、什么时候做、做到什么程度算完。你只处理系统搞不定的事。
核心转变：
人催 LLM → 等结果 → 再催 → 你设计闭环规则 → 系统自动跑 → 你只处理异常

这个转变不是概念炒作。2026 年 6 月，Peter Steinberger（OpenClaw 作者）和 Boris Cherny（Anthropic Claude Code 负责人）在 X 上同时喊出同一句话："别再手动 prompt 你的 coding agent 了。设计 loop 来帮你做这件事。" 当两家产品的核心负责人殊途同归，这不是巧合——是范式拐点已到。

2 从 Multi-Agent 到 Loop

5 月版方案的局限性

我们在 5 月 30 日讨论的 Multi-Agent 方案，核心模式是：Agent 分析 → 给你建议 → 你确认 → 执行。这套模式让"AI 帮你看数据"变成了现实，但它仍然依赖你作为决策链条的中枢——你的时间成了整个系统吞吐量的天花板。

维度5 月版：AI 辅助人决策6 月版：Loop 自动化闭环
人类角色每个决策都需要你确认你只处理异常兜底
响应速度小时级（人在岗才能处理）分钟级（7×24 自动跑）
决策依据人类的经验和直觉可量化的目标和阈值
规模上限你一天能处理的决策数Token 预算的上限
连续性你下班了它就停了晚上周末照常跑
费用控制可控（每次你手动触发）需要预设 Token 预算
适用场景战略判断、模糊决策日常运营、异常检测、策略执行

Loop Engineering 的核心机制

一个完整的 loop 包含 6 个组件。它不是 crontab——cron 执行固定脚本，loop 自己决定下一步做什么：

01
🔄 Trigger
定时 / 事件 / API 唤醒
02
📡 Sense
读数据 / 辨状态
03
🧠 Decide
LLM 决策下一步
04
⚡ Act
调用工具 / 执行修改
05
✅ Check
验证目标是否达成
06
📝 Memo
写入外部记忆后继续/停止

关键区别：cron 说"每分钟执行这个命令"，loop 说"每天凌晨检查指标，如果 DAU 跌了 10% 就下钻分析，找到原因后推给我，如果连续 3 天异常就升 P0 叫醒我"。

3 整体架构

六层架构总览

🔔 L0 · 事件触发层
⏰ Cron 定时 📨 Webhook 事件 📊 指标阈值越界 💬 人工触发
↓ 事件路由
🎛️ L1 · 编排层（Orchestrator Loop）
🎯 意图路由 📋 任务拆分 📝 记忆读写 🔄 状态管理 🔁 重试 / 升级
↓ 派发到业务 Agent
🤖 L2 · 业务 Agent 层
📊 数据分析 ⚡ 运营执行 👥 用户洞察 🔍 定位分析 💰 商业化
↓ 调用
🔧 L3 · 工具执行层
🗄️ SQL 查询 📡 API 调用 🔍 搜索 (AnySearch) 🌐 MCP Connectors 📤 推送渠道
↓ 返回结果
🧠 L4 · 持久化层
📁 HY Memory 🗃️ 状态文件 📒 Linear / Jira 📊 数据仓库
↓ 决策参考
👁️ L5 · 监控与异常层
🔴 异常升级 📈 透传面板 📋 操作日志 💰 费用看板

架构设计原则

原则 1
可验证目标驱动
每个 loop 必须有一个可量化的终止条件。没有 Check 步骤的 loop 不是 loop，是 token 熔炉。
原则 2
向上升级（Escalation）
loop 搞不定的事不硬撑。定义 P0/P1/P2/P3 等级，超出 loop 能力自动通知人。
原则 3
写入操作分级审批
🟢 读（自动）/ 🟡 建议（需确认）/ 🔴 写（二次确认+日志），分级策略不妥协。
原则 4
外部记忆是命脉
Agent 会忘，文件不会。状态、上下文、决策记录全部持久化到外部存储，不依赖 Agent 上下文。
原则 5
Token 预算硬上限
每个 loop 设 max_tokens 和 max_cost，超限自动暂停并上报。没有人想看到信用卡账单比运营收入还高。
原则 6
可审计可回滚
loop 的每个决策、每次执行都有日志。出问题了能追溯，能回滚到上一个稳定状态。

4 角色 Loop 详解

以下五个业务 Agent 各自嵌入一个独立的 loop。每个 loop 有明确的 Trigger → Sense → Decide → Act → Check → Memo 六步，都挂到编排层（L1 Orchestrator）统一管理。

📊 Agent 1：数据分析 Loop

组件定义
🔄 Trigger定时（每小时/每4小时/每日）+ 指标异常主动唤醒
🎯 Goal核心指标在安全区间，异常已识别并归因
📡 Sense拉取数仓数据：DAU / 下载 / 收入 / 留存 / 反馈量 / Crash率，对比 7 日均线和前值
🧠 Decide差值 > 15% → 自动下钻维度（渠道/版本/时段/地域）；置信度 > 85% → 输出结论；否则升到人工
⚡ Act生成异常报告 → 写入记忆 → 触发相关 Agent（如用户洞察需要看反馈趋势）
✅ Check对比目标：指标是否已恢复正常？连续 3 次异常 → P0 升级
📝 Memo写入状态文件：当前指标快照、已识别的异常列表、上次分析时间
实操例子："昨天下载量跌了 18%，Loop 自动下钻发现是某个渠道的投放素材被竞品截流。Agent 自动写了一份归因分析报告，并给商业化 Agent 发了一条"建议检查该渠道出价策略"的消息。你早上起来看到的是结论，不是原始数据。"

⚡ Agent 2：运营执行 Loop

组件定义
🔄 Trigger数据分析 Loop 发现异常 → 策略决策触发 / 每日固定检查订阅配置
🎯 Goal运营策略按配置执行完毕，效果达预期
📡 Sense读取当前策略配置、已有活动效果、资源位占用情况
🧠 Decide根据数据和规则给出建议：改投放出价？换素材？调整专题位置？自动模式或需确认
⚡ Act🟢 读：自动 → 🟡 写活动配置：需你确认 → 🔴 上架/下架：二次确认+日志
✅ Check执行后 4h 看效果：CTR/下载/收入有没有预期变化 → 有则归档 / 无则回滚
📝 Memo写入执行记录（操作人：Agent / 确认人：?) + 效果跟踪任务
关键安全设计：写入操作分三级。Agent 只能自行执行 🟢 级别的操作（如读取配置、生成报告）。任何 🟡 和 🔴 操作都生成确认卡片推送到你手机，你点"确认执行"后才落地。每个操作都有完整审计日志，可以追溯"谁在什么时候通过什么 loop 做了什么事"。

👥 Agent 3：用户洞察 Loop

组件定义
🔄 Trigger新反馈到达（实时）/ 每日汇总 / 数据分析 Loop 发现反馈量异常增多
🎯 Goal当日反馈全部分类完成 + 趋势标记 + 可操作洞察
📡 Sense抓取应用商店评论、客服工单、社区反馈、内测群消息
🧠 DecideNLP 分类（bug/建议/抱怨/咨询） → 情感分析 → 关联版本/渠道 → 趋势检测
⚡ ActP3（普通反馈）→ 自动回复模板；P2（小 bug）→ 自动比对已知问题库；P1（支付问题）→ 自动查订单；P0（闪退）→ 自动触发定位分析 Loop
✅ Check反馈量回归基线 → 归档。新增反馈趋势持续上升 → 保持警惕
📝 Memo写入反馈聚类结果 + 处理统计（自动处理 X 条 / 需要人工 Y 条）
📱 应用商店评论
差评自动回复 + 分级上报。P0 闪退类自动触发定位分析 Loop，不需要你看到差评才去查。
📋 客服工单
自动分类 + 自动处理常见问题。新出现的异常类别自动标记 + 向你推送"这个类别以前没见过"。
📊 反馈趋势
新版本发布后自动追踪反馈变化。"这个版本让好评率下降了 5 个百分点，主要集中在新功能 X 上"。

🔍 Agent 4：定位分析 Loop

组件定义
🔄 TriggerCrash 率飙升 / 用户反馈 P0 / 数据分析 Loop 发现异常指标 / 新版本发布
🎯 Goal找出根因，输出可复现步骤和修复建议
📡 Sense关联 Crash 日志、反馈文本、版本发布记录、AB 测试分组
🧠 Decide时间线交叉分析 → 关联事件 → 缩小范围 → 给出根因判断
⚡ Act生成定位报告 → 推送到开发群 → 自动开 Jira/Linear ticket（可选自动 hotfix）
✅ Check修复上线后再追踪 24h：Crash 率是否回到基线？
📝 Memo写入已知问题库 + 修复记录 + 避免再次犯同类错误的规则建议
高价值但高风险的场景：自动 hotfix 需要接入代码仓库和发布流水线。技术可行但建议只对低风险修复（配置错误、文案错误）开启自动模式，代码级别的修复至少需要人工 Code Review。

💰 Agent 5：商业化 Loop

组件定义
🔄 Trigger每日 / 收入波动 > 10% 主动触发 / 月初自动生成月报
🎯 Goal理解收入波动的原因 + 给出可落地的收入优化策略
📡 Sense拉取收入数据：广告收入 / 内购 / 付费转化漏斗 / ARPU / LTV
🧠 Decide维度下钻（渠道/版本/用户分层） → 归因 → 策略推荐
⚡ Act生成收入日报 → 推送企微 → 建议策略写入"待执行"列表 → 等待你确认
✅ Check策略执行后 48h 追踪收入变化 → 如有正向效果 → 持续执行；无效果 → 建议切换方案
📝 Memo写入收入趋势 + 已尝试策略 + 效果评估

5 编排层 · Orchestrator Loop

编排层是整个架构的"大脑"。它不是一个普通的 Agent，它管理其他所有 Agent 的生命周期：

🎯 意图路由
收到触发信号 → 判断该交给哪个 Agent。P0 事件直接触发定位分析 Loop；定时任务分发给对应 Agent；用户手动 @Agent 按自然语言路由。
📋 任务拆分
复杂任务拆成子任务分配给不同 Agent。比如"收入异常 → 数据分析 Agent 查数字 + 用户洞察 Agent 查反馈 + 商业化 Agent 出策略"。
🔄 重试与升级
Agent 执行失败 → 编排层判断重试还是升级。1 次失败重试；2 次失败换方案；3 次失败升 P0 到你。
📝 记忆管理
统一管理外部记忆。Agent A 的分析结论 → 写入记忆 → Agent B 可以读取引用。保证跨 loop 的上下文连续性。
💰 Token 预算
每个 loop、每个 Agent 有独立 Token 预算。超限自动暂停 + 通知你。防止某个 Agent 跑飞烧钱。
📊 透传面板
编排层维护"系统状态文件"：当前各 Agent 状态、最近执行记录、费用统计、待处理异常。随时查看整个体系健康度。

6 安全 · 审批 · 隔离

操作分级策略（不可妥协）

等级示例Agent 权限安全措施
🟢 读 查询数据、看报表、读配置 ✅ 自动执行 只读 DB 用户 + 无网络外发能力
🟡 建议 推荐活动策略、输出运营方案 ✅ 生成但需确认 输出 → 推送确认卡片 → 你点同意才进入待执行
🔴 写 上架/下架/改配置/调价格 ❌ 需审批 二次确认 + 完整审计日志 + 可回滚快照

隔离机制

🌲 Worktree 隔离
每个 Agent 执行写入操作时使用独立目录/分支，互不干扰。Agent A 改 AB 测试配置不会覆盖 Agent B 准备发布的活动。
📝 审计追踪
所有 Agent 操作全部写入审计日志。格式：[时间] [Agent] [操作类型] [参数] [结果/失败原因]，结构化可查询。
🔄 自动回滚
Agent 执行写操作前自动创建快照。Check 验证失败 → 自动回滚到上一个稳定状态，不影响线上服务。
🛑 一键暂停
整个体系有"一键暂停"入口。发现 loop 跑飞 → 暂停所有自动执行 → 降级为只读模式 → 人工排查后再恢复。

7 落地方案

四个阶段，从最小的闭环开始

Phase 1 · 数据分析 Loop MVP（2 周）
目标：跑通第一个完整 loop
实做：数据分析 Agent + 定时触发 + SQL 查询 + 异常检测 + 推送企微
交付：每天早上自动推送昨日运营数据简报，异常自动标红
投入：1 人 2 周（你现在的技能栈可以直接搭）
验证：你收到的数据报告准确率 > 90%
Phase 2 · 用户洞察 Loop + 反馈闭环（+2 周）
目标：反馈自动处理挂到 loop 上
实做：应用商店评论 + 客服工单接入 → 自动分类 → 自动回复（P3）→ 自动升级（P1/P0）→ 触发定位分析
交付：每天 80% 的反馈由 Agent 自动处理，只有 P0 和未知分类才到你这里
关键：反馈分类模型用 AnySearch social_media 垂直域 + LLM 分类器
Phase 3 · Loop 联动 + 运营执行（+3 周）
目标：Agent 之间自动联动，部分运营操作可一键落地
实做：编排层上线 → Agent 联动 → 🔴 操作审批流
交付："DAU 异常 → 数据分析 Loop 归因 → 运营执行 Loop 出策略 → 你确认 → 自动执行"这一完整链路跑通
关键：审批流设计必须用户友好（企微确认卡片）
Phase 4 · 商业化 + 定位分析 + 全面上线（+4 周）
目标：五个 Agent 全部跑在 loop 上
实做：商业化 Loop + 定位分析 Loop 上线 → Token 预算 + 透传面板 + 一键暂停
交付：完整的 6 层架构生产运行
验证：一个季度后统计：loop 自动处理的异常数 / 升级到你这里的事件占比 / 运营效率提升幅度

8 风险与边界

Loop Engineering 在运营场景的局限性

🎯 目标定义模糊
代码开发的"完成"可验证（测试通过），运营的"完成"往往模糊。"这个活动好不好"是参与率高好还是付费率高好？两个目标可能冲突。Loop 的终止条件必须精确，否则会变成一个无限优化的 token 熔炉。
💰 Token 成本失控
一个 loop 自己 prompt 自己、自己审查、派生子任务、不断重试……烧 token 速度可以是你手动用的 50 倍。不加预算控制就是"很有信心的 token 熔炉"。
🧪 误判成本
数据分析 Loop 把一个正常波动判定为"异常"并触发了一连串操作——这就是经典的"告警疲劳 + 自动化雪崩"。Check 步骤中的回退策略必须设计得足够保守。
🔗 系统耦合
五个 Loop 相互联动时，一个 Loop 的异常可能传染到其他 Loop。编排层的隔离机制和熔断器设计是必要的安全网。

什么场景不适合 Loops

不适合的场景原因替代方案
内容创作 / 文案撰写"好不好"完全主观，Loop 会无限改下去人工定稿，LLM 只做第一版草稿
战略方向决策需要行业经验 + 模糊判断，无法量化LLM 做信息收集 + 分析对比，决策留给人
用户隐私数据处理合规要求不允许 AI 自动处理敏感数据脱敏后给 LLM，或人工处理
