🧠问题本质：数仓是给人看的，不是给 LLM 看的

为什么让 LLM 直接读数据库做分析，成功率往往很低？问题不在模型，在数据组织方式。

😖 数仓现状：为 BI 而建
传统数仓的假设：人有领域知识，知道 dt 是分区日期、pv 是页面浏览量、ods_dwd_dws 应该从哪层开始查。

但 LLM 完全不一样：
  • 它没有「领域知识」——面对 1000 张表的复杂分层，不知道从哪张查起
  • 它不会「上下文联想」——看到 app_download_log 不知道关联 dim_app
  • 它不懂「业务口径」——「下载量」和「有效下载量」LLM 分不清
  • 它容易「盲目自信」——猜错了表名也会硬着头皮查，给错结论
✅ 解法核心：不是让 LLM 去适应你的数仓，而是改造数仓给 AI 看的接口——Schema 清晰化 + 语义视图 + 分析模板 + 口径定义 + 安全隔离，五步搞定。

🔧五步改造方案

第一步：Schema 注入——给 LLM 一张「数据地图」

这是最基础也是 ROI 最高的一步。把字段含义、取值范围、关联关系写清楚。

-- ❌ 传统建表，LLM 看不明白
CREATE TABLE app_download_log (
  app_id STRING,
  channel STRING,
  dt STRING,
  downloads INT
);

-- ✅ AI 友好建表（加 COMMENT，LLM 直接解析）
CREATE TABLE app_download_log (
  app_id STRING COMMENT '软件ID，关联dim_app.id',
  channel STRING COMMENT '下载渠道：huawei|oppo|vivo|xiaomi|official',
  dt STRING COMMENT '分区日期，格式yyyymmdd',
  downloads INT COMMENT '日下载量'
) COMMENT '软件市场每日下载日志，每app每渠道每日一条';

Schema 注入到 LLM 的推荐格式：

## 可用数据表

### 1. app_download_log（每日下载日志）
- 说明：软件市场每日下载记录，每app每渠道一行
- 字段：
  `app_id` → dim_app.id — 软件ID
  `channel` — 下载渠道：huawei|oppo|vivo|xiaomi|official
  `downloads` — 日下载量
  `dt` — 日期(yyyymmdd)

### 2. user_feedback_tb（用户反馈）
- 字段：
  `module` — 问题模块：crash|performance|feature|ui|other
  `severity` — p0|p1|p2|p3
💡 这一步的收益：LLM 看到 Schema 后，能自动构造正确的 SQL，不再瞎猜表名和字段。查询成功率从 ~30% 提升到 ~70%。
工作量：几张核心表写好 COMMENT，半天搞定。

第二步：加「AI 语义视图」——不给 LLM 裸表

建专门的 AI 查询视图，把复杂的多表关联、ETL 逻辑、口径定义封装好。

-- ❌ 不给 LLM 裸表，它不会自己 JOIN
-- ✅ 建 AI 视图，把关系写进去
CREATE VIEW ai_fact_daily_app_stats AS
SELECT
  d.app_id,
  a.app_name,
  a.category,
  d.channel,
  d.dt,
  d.downloads,
  f.feedback_count
    COMMENT '当日反馈量',
  f.crash_feedback_count,
  c.crash_rate,
  d.downloads / NULLIF(f.total_uv, 0) AS download_convert_rate
    COMMENT '下载转化率 = 下载量/详情页UV'
FROM app_download_log d
LEFT JOIN app_feedback_summary f ON d.app_id = f.app_id AND d.dt = f.dt
LEFT JOIN dim_app a ON d.app_id = a.id
LEFT JOIN app_crash_stats c ON d.app_id = c.app_id AND d.dt = c.dt;

建视图原则：

  • 视图名前缀 ai_ → 一目了然这是给 AI 用的
  • 每个字段带 COMMENT → LLM 直接读语义
  • 不暴露技术细节 → 分区字段不用展示
  • 字段名可读 → 不用 c1、t2 这种缩写

第三步：预定义「分析场景模板」——告诉 LLM 怎么查

LLM 自己从零构造 SQL 容易出错。提前定义常用的分析路径模板。

// 分析模板示例：下载量波动归因
{
  "场景": "下载量波动归因",
  "入口视图": "ai_fact_daily_app_stats",
  "分析链路": [
    {"步骤": "1.查日环比", "SQL": "SELECT dt, SUM(downloads) FROM ai_fact_daily_app_stats WHERE dt BETWEEN '{date-7}' AND '{date}' GROUP BY dt"},
    {"步骤": "2.波动>5%按渠道拆", "SQL": "SELECT channel, SUM(downloads) FROM ... WHERE dt='{date}' GROUP BY channel"},
    {"步骤": "3.渠道正常按软件拆", "SQL": "SELECT app_name, SUM(downloads) FROM ... GROUP BY app_name ORDER BY downloads"},
    {"步骤": "4.异常软件关联反馈", "SQL": "SELECT module,COUNT(*) FROM user_feedback WHERE app_id='{id}' AND dt='{date}' GROUP BY module"},
    {"步骤": "5.关联Crash数据", "SQL": "SELECT crash_rate FROM app_crash_stats WHERE app_id='{id}'"}
  ],
  "输出": "下钻结论 + 根因 + 建议"
}
💡 这一步的收益：LLM 不需要自己「发明」分析路径，按模板一步步查即可。分析质量完全可控。
工作量：3~5 个核心分析模板，写一次永复用。

第四步：加「度量语义层」——LLM 才不会搞混口径

同一个「下载量」，不同人口径不同。给 LLM 一份口径定义表。

## 指标口径定义（Metrics Dictionary）

daily_downloads
  描述: 当日软件市场所有渠道的下载总数
  口径: SUM(downloads) WHERE dt = '{date}'

download_convert_rate
  描述: 浏览详情页后实际下载的用户比例
  口径: downloads / NULLIF(detail_page_uv, 0)

crash_rate
  描述: 软件市场模块的崩溃率
  口径: crash_users / NULLIF(dau, 0)
💡 这一步的收益：LLM 不再搞混口径，生成的 SQL 准确可靠。数据分析 Agent 的结论可信度大幅提升。

第五步：权限与隔离——不让 LLM 乱跑 SQL

LLM 能分析数据，但不能写数据。

-- 创建只读用户
CREATE USER ai_analyst WITH READ_ONLY;
GRANT SELECT ON ai_schema.* TO ai_analyst;
GRANT SELECT ON ai_view.* TO ai_analyst;
REVOKE INSERT, UPDATE, DELETE, CREATE, ALTER, DROP ON *.* FROM ai_analyst;

-- 查询超时控制
SET statement_timeout = '30s';
SET work_mem = '512MB';

资源控制策略：

  • 查询超时 30 秒，防止 LLM 跑全表扫描
  • 结果集上限 1000 行，防止 Token 爆炸
  • 限制内存使用，避免占用过多资源
  • 所有查询写审计日志，可追溯

🏛️完整架构图

🔹 从数据到 AI 结论的完整链路

底层数仓 → AI 语义视图 → Schema 注入 → 分析模板 → LLM Agent → 结论+建议
ODS → DWD → DWS ai_view_daily_stats COMMENT 字段定义 下载/反馈/质量模板 DeepSeek 推理 推送 / 工单

关键设计原则

原则说明
只读优先LLM 只有 SELECT 权限，杜绝数据写入风险
语义可读字段名、视图名都要有人和 LLM 都能理解的命名
不暴露复杂度AI 视图封装多表 JOIN，LLM 只需查一个表
模板驱动分析路径预定义，LLM 只做选择+解读，不做 SQL 发明
结果可控超时/行数/资源三重限制，防止失控查询

🚀落地建议：一天搞定最小闭环

第一步：挑一张表做示范（2小时）
挑最常用的 app_download_log，把 COMMENT 补全
封装成 ai_view_daily_download 视图（关联 dim_app）
第二步：写第一个分析模板（1小时）
「下载量波动归因」模板：日环比 → 按渠道拆 → 按软件拆 → 关联反馈
第三步：接入 AI Agent（1小时）
把 Schema 注入 + 模板写入 Agent 的 system prompt
用 cron 定时跑，推送结论到企微/QQ
✅ 最小闭环跑通后：效果立竿见影，LLM 能自己查数据、下钻分析、给结论。再扩展到反馈、质量、商业化等场景，每张表花 2 小时封装，一周铺完。
