一、Claude Artifacts 的底层渲染架构
Artifacts 不是 UI 黑科技，而是一个结构化的响应协议 + 前端沙箱渲染引擎的组合。
1.1 通信协议：<antArtifact> XML 标签
Claude 在生成响应时，直接在文本流中嵌入 XML 标签。前端 UI 解析响应流，提取 <antArtifact> 标签内容渲染到侧边面板。不改变底层 API，只在响应内容上做标记。
<antArtifact identifier="metrics-dashboard" type="application/vnd.ant.react" title="Dashboard"> // 完整的组件代码... </antArtifact>
1.2 六种 Artifact 类型与 MIME 体系

1.3 HTML/React 的沙箱渲染流程
安全限制：无网络请求 · 无文件系统 · 无持久存储 · 仅允许 cdnjs 加载脚本
React 运行时：Babel standalone 浏览器端转译 JSX · 内置 React/Recharts/Lucide/shadcn/ui

1.4 identifier 版本控制机制
每个 artifact 有 kebab-case identifier。用户说"修改这个"时，Claude 复用 identifier 更新已有 artifact，而不是新建。前端根据 identifier 判断新建还是替换，实现自然的迭代体验。
// 首次创建 <antArtifact identifier="factorial-script" type="application/vnd.ant.code" ...> // 用户说"改成支持负数" → 复用 identifier <antArtifact identifier="factorial-script" type="application/vnd.ant.code" ...>

二、风格统一是如何做到的
通过系统提示词硬性约束技术选型，而不是靠模型自己"学会审美"。
2.1 System Prompt 中的样式规则
// 逆向出来的系统提示词样式约束 · Use Tailwind classes for styling. · DO NOT USE ARBITRARY VALUES (e.g. h-[600px]). · lucide-react@0.263.1 is available for icons. · recharts is available for charts. · shadcn/ui components are importable. · cdns.cloudflare.com is the only allowed CDN. · Use placeholder images: <img src="/api/placeholder/400/320" />
关键洞察：风格统一不是模型"学会了设计审美"，而是系统提示词强制指定了技术栈，自然保证了输出一致性。
2.2 为什么选 Tailwind？
1. 原子化类名降低幻觉 — 写 flex items-center 比手写 display: flex; align-items: center; 更不容易出错。

2. 固定词表 — Tailwind 的类名是有限集合，模型不需要"发明"CSS 属性值。

3. 禁用 Arbitrary Values 的意义 — 强制使用预设设计系统，避免生成奇怪的尺寸/颜色。
❌ 无约束时模型可能写： <div style="margin-left: 17.3px; background: #A7C8E9; font-size: 0.92rem"> （随机值，没有一致性） ✅ Tailwind + 禁止任意值： <div class="ml-4 bg-blue-100 text-sm"> （设计系统内的固定值，视觉一致）
2.3 内置组件生态
shadcn/ui 提供统一卡片、提示框、按钮等组件；Lucide 提供统一线条图标风格。模型只需要 import 这些组件，不再从零"发明"UI。
import { Card, CardHeader, CardContent } from '@/components/ui/card' import { Camera, User } from 'lucide-react' const ProfileCard = () => ( <Card> <CardHeader>User Profile</CardHeader> <CardContent><User /> John Doe</CardContent> </Card> )
2.4 代码结构约束
· 单文件 — HTML/JS/CSS 必须在一个文件内
· 禁止占位符 — 不能写 "// rest of the code remains the same"
· 默认导出 — 统一组件导出规范
· 无 required props — 所有组件拿来即用

三、如何判定用户 query 需要生成 artifact
双层评估机制：显式规则 + 隐式 chain-of-thought。
3.1 显式规则：应该 / 不应该

✅ 应该创建

  • 内容量>15 行，足够 substantial
  • 可迭代用户可能会修改或迭代
  • 自包含脱离对话也能独立理解
  • 外部可用可用于对话之外的场景
  • 可引用会被多次引用

❌ 不应该创建

  • 简短简单信息、短代码片段
  • 解释性主要为了阐明概念
  • 评论对已有 artifact 的反馈
  • 依赖上下文脱离对话就无意义
  • 一次性用户不可能修改
  • 一句话看起来是一次性询问
3.2 <antThinking> 隐式预评估
最巧妙的设计——系统提示词要求 Claude 在生成 artifact 前，在 <antThinking> 标签中思考一句话：
"当前内容是否符合标准？如果不用 artifact 是否也能很好呈现？"

这个标签在服务端被剥离，用户永远看不到。但它的存在起到了 chain-of-thought 效果，让模型自检。
实际例子（来自逆向工程）： <antThinking>Drawing a simple blue circle in SVG is a good candidate for an artifact. It's a standalone visual element that can be reused.</antThinking> 另一个例子（判定不创建）： <antThinking>The user asked about Python list methods. Putting this in an artifact would make it harder to scan and ask follow-ups.</antThinking>
3.3 设计哲学：倾向于不创建
"If unsure whether the content qualifies as an artifact... err on the side of NOT creating an artifact."

"Prefer in-line content (don't use artifacts) when possible. Unnecessary use of artifacts can be jarring for users."

这是一个精心设计的保守策略——artifact 打断对话流，滥用会破坏用户体验。

四、对我们的启示与可借鉴之处
Claude Artifacts 不是黑盒，而是一个可分析的系统设计。我们可以从中借鉴多层模式。
4.1 协议设计：文本流中嵌入富媒体
Claude 用 XML 标签在文本流中标记结构化内容，不改变底层 API。这实际上是一种流式富媒体协议。

我们在做的类似的事：每次让你上传 HTML 报告到 GitHub Pages，本质就是"在文本对话中嵌入富媒体"——我给你内容，你给我 URL。这已经是我们版本的 artifact 协议。
4.2 风格统一：建立自己的 HTML 模板系统
我们目前每次写 HTML 报告虽然视觉统一（暗色主题、卡片布局），但还没有正式的模板系统。可以借鉴 Claude 的做法：

· 固定 CSS 框架 — 已有的设计系统（渐变背景、圆角卡片、暗色主题）可以收敛为一个基础模板
· 限定设计 token — 主色、间距、字体等固定下来，不再每次手写
· 组件库（可选） — shadcn/ui 那样的卡片、表格、标签组件可以做成 CSS 类库
// 当前我们的模板核心样式： -- 渐变色头: gradient(135deg, #0d0d2b, #1a0a3e, #2d1b69) -- 卡片: rgba(255,255,255,0.06) 背景 -- 标签: rgba(81,207,102,0.15) 绿色标签 -- 代码块: rgba(0,0,0,0.35) -- 圆角: 14px/20px/24px -- 字体: -apple-system, PingFang SC, sans-serif // 这些可以固化成一个 base.css 或模板
4.3 触发判定：开与关的决策框架
我们生成 HTML 报告时也可以采用类似的双层评估：

应该生成 HTML 报告：
· 需要多 Tab 组织的复杂内容
· 含有表格、图表、代码块等结构化信息
· 需要比对话更丰富的视觉呈现
· 可能被多次查阅或分享

不应该生成 HTML 报告：
· 短信息（一两段话就能说清楚）
· 实时问答（你在等答案的场景）
· 简单建议（"买还是卖"这类）
· 一次性话题，不会再回看
实际上你已经在用这个框架了——恒科分析、Karpathy、Artifacts 这些深度内容都生了 HTML 报告。点到为止的问答都在对话里解决。
4.4 版本控制：GitHub Pages 作为 artifact 存储
Claude 用 identifier 管理 artifact 版本迭代。我们用的 GitHub Pages + 子目录 本质上也是版本控制——

· 每个报告是一个独立 URL
· 更新时覆盖 index.html 即可
· git 历史自动保留所有版本

可以进一步优化：在首页（songyafei.github.io）自动列出所有报告，类似 Claude 的 artifact 侧边栏。
