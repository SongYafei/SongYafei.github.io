🔍 Karpathy 观察到的三大失败模式
模式描述典型例子
1. 沉默假设AI 替你做了错误的假设，然后默默跑下去了——不澄清、不质疑、不展示取舍你说了"A"，AI 以为你要"B"，直接写了 500 行实现
2. 过度工程AI 喜欢把代码搞得很复杂，过度抽象，100 行能解决的问题写 1000 行对两个字段做"企业级"抽象工厂模式
3. 附带损害改代码时碰了不该碰的地方，删了注释、改了格式、动了无关代码让你加个按钮，它把你的整个 CSS 重构了一遍
📐 四大核心原则
Forrest Chang 让 AI 把 Karpathy 的观察提炼成 actionable 的指令，又让 AI 自审精简到 65 行。
1️⃣ Think Before Coding — 先想再写
核心：写代码前必须显式陈述假设，如果有多种解读，全部列出来；不清楚就停下来问。

解决了：沉默假设问题。

好例子 vs 坏例子：
❌ AI 默默地选了一个方案
"Adding email validation..."
→ 手写了一个简单的正则匹配

✅ AI 先展示取舍
"I see two approaches:
Option A: simple regex for basic validation
Option B: RFC 5322 compliance with DNS checking
Which fits your needs?"
2️⃣ Simplicity First — 简单优先
核心：写最少的代码解决当前问题。没有要求的特性不要做，没有要求的抽象不要加，没有要求的灵活性不要留。

自检标准：如果 200 行能写成 50 行，重写。问自己"资深工程师会说这过度复杂吗？"

解决了：过度工程问题。
❌ AI 的过度抽象
class ValidationStrategyFactory {
  // 200 行的抽象工厂，只为了验证 2 个字段
}

✅ 简单直接
function validateEmail(email: string): boolean {
  // 20 行，看得懂，改得了
}
3️⃣ Surgical Changes — 手术刀式修改
核心：只动必须动的代码。不要"顺手改进"旁边的代码、注释或格式。不要重构没坏的东西。匹配现有风格（即使你会用另一种写法）。

关键区分：你的改动造成了无用的 import/变量 → 删除（你的锅）。已有的死代码 → 只提一句，不删（除非被要求）。

检验标准：每一处改动都应能直接追溯到你的需求。

解决了：附带损害问题。
❌ AI "顺手"改了一堆
+ // 你要的功能（1 行）
+ // 重命名了 3 个变量
+ // 改成了箭头函数
+ // 删了 2 行注释（删除无误但也没人要求）

✅ 手术刀式
+ // 你要的功能（1 行）
// 别的什么都没碰
4️⃣ Goal-Driven Execution — 目标驱动执行
核心：这是 Karpathy 最关键的洞见——"LLMs are exceptionally good at looping until they meet specific goals."

不要告诉 AI 怎么做，给它成功标准，让它自己去迭代。

转化方式：
❌ 指令式（弱标准）
"Add validation"
"Fix the bug"
"Make it work"

✅ 声明式目标（强标准）
"Write tests for invalid inputs, then make them pass"
"Write a test that reproduces the bug, then make it pass"
"Ensure tests pass before and after refactoring"

多步骤任务规划：
1. [Step] → verify: [check]
2. [Step] → verify: [check]
为什么有效：强成功标准让 AI 可以自主循环迭代。弱标准（"搞好"）需要不断追问，既浪费时间又容易跑偏。

# CLAUDE.md

Behavioral guidelines to reduce common LLM coding mistakes.
**Tradeoff:** These guidelines bias toward caution over speed.
For trivial tasks, use judgment.

## 1. Think Before Coding

**Don't assume. Don't hide confusion. Surface tradeoffs.**

Before implementing:
- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them - don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, stop. Name what's confusing. Ask.

## 2. Simplicity First

**Minimum code that solves the problem. Nothing speculative.**

- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" or "configurability" that wasn't requested.
- No error handling for impossible scenarios.
- If you write 200 lines and it could be 50, rewrite it.

Ask yourself: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

## 3. Surgical Changes

**Touch only what you must. Clean up only your own mess.**

When editing existing code:
- Don't "improve" adjacent code, comments, or formatting.
- Don't refactor things that aren't broken.
- Match existing style, even if you'd do it differently.
- If you notice unrelated dead code, mention it - don't delete it.

When your changes create orphans:
- Remove imports/variables/functions that YOUR changes made unused.
- Don't remove pre-existing dead code unless asked.

The test: Every changed line should trace directly to the user's request.

## 4. Goal-Driven Execution

**Define success criteria. Loop until verified.**

Transform tasks into verifiable goals:
- "Add validation" → "Write tests for invalid inputs, then make them pass"
- "Fix the bug" → "Write a test that reproduces it, then make it pass"
- "Refactor X" → "Ensure tests pass before and after"

For multi-step tasks, state a brief plan:
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]

Strong success criteria let you loop independently.
Weak criteria ("make it work") require constant clarification.

---

These guidelines are working if:
fewer unnecessary changes in diffs,
fewer rewrites due to overcomplication,
and clarifying questions come before implementation rather than after mistakes.
⚡ 如何应用到你的开发流程
你是 OpenClaw 用户 + AI Agent 重度使用者。Karpathy 的四大原则可以直接迁移到你和 AI 的协作中。

📌 原则 1：Think Before Coding → 让 AI 先说话再动手

给 AI 下达任务时，加一句 "先说说你的理解和方案，确认后再说"。

你的场景：让 OpenClaw 执行复杂操作（部署、改配置、写脚本）前，让 AI 先列计划和假设，避免直接动手后偏离预期。

✅ "帮我分析这个 issue，先说你的理解和修复方案"
✅ "列出你认为可能出坑的地方"

📌 原则 2：Simplicity First → 明确要求"别过度设计"

对 AI 输出的期望要明确："最简单的方案，最小代码量"。

你的场景：生成脚本/配置时，AI 喜欢加额外的参数、冗余的 fallback、不必要的抽象。

✅ "用最直接的方式写，不要抽象，不要考虑未来的扩展"
✅ "这个脚本只做一个事，别加额外功能"

📌 原则 3：Surgical Changes → 改哪动哪

修改现有代码/配置时："只改我指定的地方，不要改动其他任何内容"。

你的场景：修改 gateway.json、openclaw.json、Python 脚本等配置文件时，AI 经常会顺手格式化 or 调整不相关的地方，造成不必要的 diff。

✅ "只添加这个字段，别动已有配置的任何格式"
✅ "不要重新格式化文件，保持原样"

📌 原则 4：Goal-Driven → 给验证标准，而不是操作步骤

这是最关键的迁移。

❌ "帮我配置一下 Nginx"（AI 可能过度配置了一堆你不需要的东西）
✅ "Nginx 转发 /api 到 localhost:3000，验证：curl localhost:8080/api/health 返回 200"

你在 OpenClaw 中的最佳实践：
✅ "执行这个命令，验证：xxx 日志中出现 yyy 字样"
✅ "修改 config，验证：执行 curl 检查确认生效"

🛠️ 实操：给你自己的项目加 CLAUDE.md
在你的项目根目录创建一个 CLAUDE.md 或 AGENTS.md（OpenClaw 已支持），写入项目特定的规则：
# PROJECT GUIDELINES

## 编码规范
- 优先 Python 3.11+，不要用兼容 2.x 的写法
- 错误处理：使用 try/except，不要用 if-return 模式
- 日志使用 logging 模块，不要 print

## 配置修改规范
- 修改 openclaw.json 请使用 config patch 命令
- 不要直接编辑 config 文件格式，以免破坏 JSON5 注释

## Karpathy 规则
1. Think Before Coding — 动手前列出假设和方案
2. Simplicity First — 最小代码量
3. Surgical Changes — 只改指定范围
4. Goal-Driven — 给出验证标准
